# UI_SPEC

## 1. 文档目标

本文档用于冻结“井周智修”Demo 的 UI 方向、页面结构、设计 token、组件复用规则与 Figma-to-code 交付约束。

适用范围：
- `apps/web` 的 MVP 路由与核心交互；
- `packages/ui` 的基础组件与少量领域组件；
- 无 Figma 文件、无 Figma MCP 连接前提下的设计系统落地规则；
- 面向客户路演的桌面优先工作台界面，同时保证平板与手机不破版。

不包含：
- 业务代码；
- API 设计；
- 实际 Figma 画布写入；
- 品牌营销页或官网式首屏设计。

## 2. 产品体验目标

界面必须同时传达五种感受：
- 市政道路行业感：像工程系统，不像互联网营销页。
- AI 诊断科技感：地下病害被看见，诊断过程可信且可解释。
- 工程数据可信感：指标、阈值、图例、时间线、工艺参数清楚。
- 修复前后对比冲击力：从“为什么修”直接过渡到“修完怎样验证”。
- 非技术人员可理解：领导不需要懂雷达，也能看懂风险、方案、验收结果。

## 3. 总体设计原则

### 3.1 工作台优先
- 采用密集但清晰的工作台式布局，优先信息层级和操作效率。
- 不做大面积留白的落地页结构，不做全屏海报式 Hero。
- 页面以“页头 + 主内容 + 次级侧栏/底栏”组织，避免卡片套卡片。

### 3.2 工程解释优先于视觉炫技
- 所有颜色都必须有业务含义，尤其是风险色、状态色、图层色。
- 地图、热力图、施工模拟、验收报告必须先服务理解，再服务美观。
- 不使用装饰性渐变球、霓虹描边、玻璃态等脱离行业语境的视觉。

### 3.3 一眼看懂闭环
- 每个主页面都要明确当前环节、上一环节证据、下一环节动作。
- 任何页面停留 5 秒内，用户都应能回答“这是哪一座井”“当前风险是什么”“下一步要干什么”。

### 3.4 数据对比优先
- 关键结论必须配对出现：修复前/修复后、风险高/风险低、异常区/正常区、建议值/实时值。
- 不用单独的“AI 很强”文案来撑科技感，直接用检测图层、诊断过程和参数回放来体现。

### 3.5 文本与图形不重叠
- 图表标题、图例、数值标签必须预留固定区域。
- 表格单元格与标签组件必须定义截断规则，不允许压缩到重叠。
- 移动端不保留桌面端并排结构，优先改为纵向堆叠。

## 4. 视觉系统

### 4.1 视觉方向
- 基底来自沥青、钢构、道路标线、施工标识。
- 中性面偏深灰蓝与混凝土灰，强调专业与耐看。
- 风险与诊断信息使用高辨识信号色，但只在关键区域集中出现。
- AI 相关区域允许使用有限的青蓝色扫描线、网格线、轮廓描边，避免全局科技蓝泛滥。

### 4.3 色彩使用规则
- 风险色只用于风险、病害等级、施工报警、验收结论，不泛化到普通按钮。
- 普通主按钮使用深蓝灰或深青灰，不直接复用红/橙风险色。
- 热力图必须永远配套图例和深度/置信度说明，不允许脱离图例独立出现。
- 地图默认底色保持克制，避免与 marker/heatmap 抢夺注意力。

### 4.4 字体与数字样式
- 中文正文：`Noto Sans SC`。
- 工程数值、时间、分贝、毫米、注浆量：`Barlow Semi Condensed`。
- 标题不使用夸张字重；以字重和密度建立秩序，不依赖大字号堆气势。

推荐字号：

| Token | 建议值 | 用途 |
| --- | --- | --- |
| `--text-hero` | 28/36 | 页面总标题，仅工作台首页使用 |
| `--text-h1` | 24/32 | 页面标题 |
| `--text-h2` | 20/28 | 区块标题 |
| `--text-h3` | 16/24 | 子模块标题 |
| `--text-body` | 14/22 | 常规正文 |
| `--text-small` | 12/18 | 辅助说明 |
| `--text-mono` | 13/20 | 参数、时间轴、日志 |

规则：
- KPI 数值使用等宽数字或半等宽数字风格。
- 单位与数值分离排版，数值高对比、单位低对比。
- 表格正文最小不低于 12px。

### 4.5 间距、圆角、阴影
- 4px 为基础单位。
- 页面主栅格：24px。
- 面板内边距：16px / 20px / 24px 三档。
- 面板圆角：`12px`；轻量标签/输入：`8px`；报告纸张容器：`16px`。
- 阴影克制，仅用于浮层、抽屉、地图悬浮信息窗。
- 不使用大面积悬浮卡片阴影堆层次，优先靠背景深浅和边框区分。

## 5. 全局布局

### 5.1 应用 Shell
- 顶部全局栏：56px 高。
- 左侧主导航：桌面端 72px 图标栏，展开态 220px。
- 主内容区：最小宽度按 1280px 桌面工作台设计。
- 页面内部遵循“标题区 -> 操作区 -> 主内容区 -> 补充信息区”。

顶部全局栏包含：
- 产品名与状态标识；
- 全局道路/区域筛选；
- 演示模式切换；
- 当前时间/数据版本；
- 帮助入口。

左侧导航顺序：
1. `Dashboard`
2. `Map`
3. `Manhole Detail`
4. `Diagnosis`
5. `Plan`
6. `Simulation`
7. `Acceptance`
8. `Demo Script`

### 5.2 页面头部模式
每个页面统一包含：
- 页面标题；
- 当前井编号与道路名；
- 风险等级与评分；
- 上一步/下一步导航；
- 数据更新时间；
- 当前页面的主 CTA。

### 5.3 面板组织规则
- 单页最多一个“主视觉面板”，例如地图、热力图、施工剖面或报告预览。
- 其余信息用同层级信息块并列，避免大卡片里再塞小卡片。
- 数据条、标签组、对比条带可直接嵌入面板，不额外再包卡片。

## 6. 路由级布局规范

### 6.1 `/dashboard`

目标：
- 5 秒内建立“这是一套道路检查井风险管理与诊修平台”的认知。
- 支撑领导视角的总览和技术视角的异常聚焦。

布局：
- 第一行：总览标题、区域筛选、时间维度、演示入口。
- 第二行：4 个核心 KPI 横向排布。
  - 高风险井数量
  - 本周待处置数量
  - 平均开放交通时间
  - 修复后复检达标率
- 第三行：左侧风险分布图，中间处置进度图，右侧高风险井清单。
- 第四行：近期施工/验收动态时间轴。

关键交互：
- 点击高风险井清单行直接跳转对应 `/manholes/[id]`。
- 点击风险等级图例联动过滤整页内容。
- 路演模式按钮进入 `/demo-script`，并记住当前选中井。

状态要求：
- Loading：KPI 骨架屏先出现，图表区域显示“正在汇总检查井数据”。
- Empty：展示“暂无井档数据，请先加载 seed 数据”，附返回地图按钮。
- Error：在图表位内展示重试，不用全屏错误覆盖整个工作台。

焦点顺序：
1. 区域筛选
2. 时间维度
3. 路演模式
4. KPI 区
5. 图表
6. 高风险井清单

### 6.2 `/map`

目标：
- 作为核心入口页，快速定位高风险检查井并触发闭环。

布局：
- 左侧 320px 风险列表与筛选器。
- 中央主区域为地图。
- 右侧 360px 井点详情抽屉，默认收起。
- 地图右下固定图例、缩放、图层开关。

地图规则：
- Marker 形状用“圆环 + 中心实点”，表达井口而不是普通 POI。
- 颜色映射 A/B/C/D 风险等级。
- 被选中 marker 增加白色外圈与脉冲描边，不使用夸张发光。
- 聚合态显示数量与主导风险色。

关键交互：
- 点击 marker：打开右侧抽屉，显示井基础信息、缩略照片、评分、上次维修时间、进入详情按钮。
- 键盘 Tab 可到达 marker 列表替代入口；地图本体不作为唯一交互路径。
- 风险筛选、道路类型筛选、是否已诊断筛选必须实时作用于地图与左侧列表。

空/错态：
- 地图失败时保留左侧列表可用，右侧展示“地图加载失败，可从列表进入井档”。
- 无结果时显示“当前筛选条件下无检查井”并提供清除筛选。

焦点顺序：
1. 筛选器
2. 检索输入
3. 左侧井列表
4. 地图图层开关
5. 已打开的右侧抽屉动作按钮

键盘说明：
- 左侧列表上下方向键切换高亮行。
- `Enter` 打开详情抽屉。
- 抽屉内 `Enter` 可触发“进入详情”。

### 6.3 `/manholes/[id]`

目标：
- 用一页说明这座井为什么被关注，并给出诊断入口。

布局：
- 顶部：井编号、道路名、所属管线、风险评分、病害等级、上次维修时间。
- 中区三栏：
  - 左：现场照片与缩略图库
  - 中：关键检测指标、趋势、历史维修
  - 右：雷达预览剖面与井周结构示意
- 底部操作带：
  - `开始AI诊断`
  - `查看原始雷达`
  - `查看历史维修`
  - `生成维修方案`

关键交互：
- 指标悬停/聚焦时联动右侧结构示意高亮对应异常区域。
- 点击照片缩略图打开灯箱，保留拍摄时间、角度、备注。
- 诊断前如果用户直接点“生成维修方案”，应提醒需先完成诊断或使用已有诊断结果。

状态要求：
- 缺少照片时显示占位说明，不隐藏图片区。
- 缺少历史维修记录时显示“暂无历史维修，不代表无历史病害”。

### 6.4 `/manholes/[id]/diagnosis`

目标：
- 这是最重要的演示页，必须让用户看见地下病害并理解推荐原因。

布局：
- 左侧 280px：诊断流程、输入数据、诊断日志。
- 中央：诊断主画布。
  - 默认展示井周平面热力图与剖面切换。
  - 支持查看 polygon 叠层、深度分层、风险区域说明。
- 右侧：诊断结论、病害分区表、推荐工法摘要。

诊断主画布样式：
- 平面图使用道路俯视语义，井口位于中心，异常区围绕井周扩散。
- 剖面图使用结构层分层色块，显示沥青面层、基层、路基、井筒边界。
- 热力图不使用彩虹色，采用“正常蓝灰 -> 松散黄橙 -> 脱空红 -> 深层禁入深红”的单向风险渐进。
- polygon 区域边界必须可见，边界线比填充更重要。

关键交互：
- `开始诊断` 后，进度文案按步骤出现：
  - 识别井周裂缝
  - 分析雷达异常反射
  - 判断基层密实度
  - 匹配历史病害
  - 生成维修建议
- 诊断完成后自动把焦点移动到“诊断结果标题”，并通过 `aria-live="polite"` 播报。
- 平面/剖面切换为分段控件，支持方向键切换。
- 点击病害分区表某行，主画布聚焦对应区域。

反馈时序：
- 诊断启动后 150ms 内进入运行态。
- 若结果返回超 1.2s，显示分步进度，不用单一 loading spinner。
- 完成后 600ms 内显示总结果，再依次显示分区表与推荐理由。

错误与回退：
- 若诊断失败，保留最近一次成功结果并提示“当前结果未更新”。
- 若无 polygon，仅有热力栅格，需明确说明“当前仅输出热力图数据”。

### 6.5 `/manholes/[id]/plan`

目标：
- 把诊断结论翻译成可执行工法、材料和参数。

布局：
- 左：诊断摘要卡与推荐理由。
- 中：工法组合与施工参数主表。
- 右：材料清单、工期、开放交通时间、减废收益。
- 底：退出微创条件与风险提示。

内容结构：
- 推荐工法主卡：`微孔注浆 + 井座锁固 + 快硬材料恢复`
- 参数表：
  - 孔位数量
  - 孔深范围
  - 注浆量范围
  - 表层修复面积
  - 施工时间
  - 开放交通时间
- 材料表：
  - 材料名
  - 用途
  - 推荐用量
  - 注意事项

关键交互：
- 点击参数项，在右侧示意图高亮对应孔位或工序。
- 如果等级为 D，页面主结论切换为“建议复核/退出微创工法”，按钮改为“查看复核建议”。

### 6.6 `/manholes/[id]/simulation`

目标：
- 清晰展示微创施工不是大开挖，而是受控、分步、可监测的过程。

布局：
- 上方：四步工序 Stepper。
- 中央：施工剖面主画布。
- 右侧：实时施工参数面板。
- 下方：前后对比条带。

工序固定为四步：
1. 精准布孔
2. 低压分级注浆
3. 井座锁固与调平
4. 快硬材料恢复

主画布样式：
- 使用半剖透视或正剖面，不做复杂 3D 漫游。
- 地下空洞、松散区、浆液扩散、井座、表层恢复要分层可见。
- 当前步骤元素使用高亮描边，其余元素降饱和。

实时参数面板：
- 注浆压力
- 当前孔注浆量
- 累计注浆量
- 路面抬升监测
- 异常报警

交互与反馈：
- Stepper 可点击跳步回看。
- 自动播放时每步 4-6 秒；手动模式下保留“上一工序/下一工序”。
- 报警值超过阈值时只高亮对应参数，不让整个页面闪烁。

无障碍：
- 自动播放默认可暂停。
- 所有步骤变化需在旁侧日志区输出文本，避免只靠动画理解。

### 6.7 `/manholes/[id]/acceptance`

目标：
- 以工程验收口径证明“修完了且可追溯”，不是只展示漂亮结果。

布局：
- 顶部：验收结论条，显示是否满足开放交通条件。
- 主体左右对比：
  - 左为修复前
  - 右为修复后
- 下方：
  - 关键指标对比表
  - 复检建议
  - 一井一档摘要
  - 导出报告入口

关键对比指标：
- 井盖与路面高差
- 三米直尺平整度
- 声振峰值
- 脱空区/热力图
- 封路时长
- 建筑垃圾减少比例

话术约束：
- 使用“验收状态下无明显跳动、空响和异常冲击声”。
- 不使用“彻底消除噪音”“永久不再复发”。

交互：
- 点击对比指标行，联动上方前后图与对应解释。
- 导出前提供“预览报告”而不是直接下载。

### 6.8 `/demo-script`

目标：
- 帮销售/技术路演使用固定节奏展示全流程。

布局：
- 左：演示章节导航。
- 中：当前章节画面嵌入。
- 右：讲解词、建议停留时间、需要强调的数据点。

章节顺序固定：
1. 平台总览
2. 高风险井定位

## 7. 组件清单与复用策略

### 7.1 优先复用的基础组件
- `Button`
- `Input`
- `Select`
- `Tabs`
- `Dialog`
- `Drawer`
- `Tooltip`
- `Table`
- `Badge`
- `Progress`
- `Separator`
- `Skeleton`

按钮图标优先使用 `lucide-react`，不新增额外图标包。

### 7.2 建议新增的领域组件

| 组件名 | 必要性 | 用途 |

| `ConstructionStepper` | 必需 | 施工四步切换 |

控制范围：
- 除上述领域组件外，不新增大而全的自定义容器体系。
- 卡片容器尽量由 `div + token` 实现，不再包一层通用 `Card` 套 `Card`。

### 7.3 组件状态规范
- `Button`：默认、悬停、按下、禁用、加载。
- `Badge`：必须支持风险态、信息态、通过态、报警态。
- `Table`：支持行 hover、选中、高亮联动、空态。
- `Drawer`：必须支持键盘关闭和焦点回退。
- `Tabs`：切换时保留高度稳定，避免主画布抖动。

## 8. 地图、热力图、施工模拟、报告样式规范

### 8.1 地图样式
- 采用低饱和底图，避免真实卫星图抢信息。
- 道路层淡灰，重要道路用浅米黄或浅蓝灰描边。
- Marker 数量多时优先做聚合和列表联动，不靠密集文字标签。
- 图例固定显示四档风险和“未诊断”。

### 8.2 诊断热力图样式
- 井口居中，周边 1.5m-2.0m 为主要展示范围。
- 必须同时展示：
  - 风险色图层
  - 图例
  - 深度信息
  - 置信度说明
- 热力图 opacity 不超过 72%，保证底层结构边界仍可识别。

### 8.3 施工模拟样式
- 优先“工程剖面图 + 动态材料流向”。
- 不做游戏化粒子特效。
- 每一步要有简短结论，例如“当前孔注浆完成，累计 38L，抬升 0.6mm，未超阈值”。

### 8.4 报告样式
- 网页内预览采用 A4 比例纸张容器，背景白色，外围为浅灰工作台。
- 报告页头包含项目名、井编号、道路名、报告时间、二维码占位。
- 报告章节固定：
  1. 基本信息
  2. 修复前病害证据
  3. AI 诊断结果
  4. 热力图/剖面图
  5. 注浆孔位与过程数据
  6. 材料与工艺
  7. 验收结论
  8. 复检建议
- 报告内表格使用高对比表头，不使用花哨底纹。

## 9. 图标与图表建议

### 9.1 图标建议
- 地图定位：`MapPinned`
- 风险告警：`TriangleAlert`
- AI 诊断：`ScanLine`
- 雷达/剖面：`Waves`
- 维修方案：`Wrench`
- 施工模拟：`Drill`
- 验收报告：`FileCheck`
- 历史档案：`FolderKanban`
- 时间线：`Clock3`

规则：
- 同一屏不混用描边粗细明显不同的图标。
- 图标只辅助识别，不承担唯一语义。

### 9.2 图表建议
- 风险分布：堆叠柱图或环形图。
- 趋势变化：折线图。
- 病害区域占比：水平条图。
- 施工实时参数：小型折线 + 当前值面板，不用仪表盘。
- 前后对比：双列对比条带或哑铃图。

不建议：
- 3D 饼图；
- 复杂雷达图堆太多维度；
- 没有工程意义的动态仪表盘。

## 10. 状态设计与反馈节奏

### 10.1 Loading
- 首屏数据：先骨架，再数字，再图表。
- 诊断与报告生成：显示步骤化进度，不用无限旋转等待。
- 地图：优先出现底图与列表，再补 marker。

### 10.2 Empty
- 每个空态都要说明为什么空、能做什么。
- 示例：
  - “暂无已诊断结果，可先发起 AI 诊断”
  - “当前井无历史维修记录”
  - “当前筛选条件下无高风险井”

### 10.3 Error
- 错误在局部容器内处理，除非整页无法工作。
- 必须给重试或替代路径。
- 技术错误文案不直接暴露接口堆栈。

### 10.4 Success
- 诊断完成、方案生成、报告生成都要有明确成功反馈。
- 成功反馈优先放在结果区域标题处，不弹出打断式 toast 作为唯一提示。

## 11. 响应式与可访问性

### 11.1 断点策略
- `>=1440px`：完整三栏工作台。
- `1280-1439px`：保留三栏，但侧栏可缩窄。
- `768-1279px`：两栏或上下堆叠，地图与主画布优先。
- `<768px`：不保留复杂三栏；采用分段信息流与底部操作条。

### 11.2 响应式规则
- 地图页移动端默认展示“列表优先，地图次之”的切换视图。
- 诊断页移动端默认先显示结论，再显示热力图。
- 报告页移动端只做浏览，不作为主要导出入口。

### 11.3 对比度与可读性
- 正文与背景对比不低于 WCAG AA。
- 风险色标签必须配文字，不只靠颜色区分。
- 热力图上文字标签应放在独立信息区，不直接压在高噪声色块上。

### 11.4 键盘与焦点
- 所有按钮、筛选器、抽屉、分段切换都必须可键盘到达。
- 焦点样式使用 2px 高对比描边，不只靠阴影。
- 抽屉/弹层打开时聚焦首个可操作项，关闭后返回触发按钮。
- 地图 marker 需有列表等效入口，避免地图成为无障碍死角。

### 11.5 动效与减弱策略
- 仅在诊断进度、施工步骤、前后对比时使用关键动效。
- 尊重 `prefers-reduced-motion`，此时动画改为淡入和状态切换。

## 12. Figma-to-Code 规则

说明：当前项目未提供目标 Figma 文件或 MCP 连接，因此本节输出可直接用于后续建画布和编码的规则，而不是实际 Figma 操作记录。

### 12.1 Figma 文件页建议
- `00 Foundations`
- `01 Components`
- `02 Dashboard & Map`
- `03 Diagnosis Flow`
- `04 Plan & Simulation`
- `05 Acceptance & Report`
- `06 Demo Script`

### 12.2 变量与样式规则
- 颜色、间距、圆角必须全部 token 化。
- Figma 变量命名与代码 token 一致，例如：
  - `color/risk/a`
  - `color/diag/void`
  - `space/16`
  - `radius/panel`
- 不直接在 Figma 图层或代码里写散落的 hex。
- 文本样式按 `heading/page`, `heading/section`, `body/default`, `data/metric`, `caption/default` 建立。

### 12.3 组件规则
- Figma 组件与代码组件一一对应，优先覆盖：
  - `Button`
  - `Badge`
  - `Tabs`
  - `Drawer`
  - `Table`
  - `RiskBadge`
  - `MetricStrip`
  - `ConstructionStepper`
- 组件应通过 variant 表达状态，不复制独立图层改颜色。
- 图标位作为 slot，不把图标烘焙进按钮底图。

### 12.4 画布规则
- 所有页面框架使用 Auto Layout，不手工堆坐标。
- 桌面端主画布宽度：
  - 页面帧：1440
  - 内容区最大宽：1360
- 单屏必须体现真实数据密度，不做“为了好看而减信息”的虚假版式。
- 页面帧命名按路由命名，如 `route/map/default`、`route/manholes-id/diagnosis-ready`。

### 12.5 代码映射规则
- `packages/ui` 存放基础与领域组件。
- `apps/web` 页面只负责组装，不复制 token 和复杂样式。
- 颜色通过 CSS variables 或 Tailwind theme token 暴露，不写内联色值。
- 图标统一来自 `lucide-react`。
- 复杂数据图层组件建议命名：
  - `MapRiskWorkbench`
  - `DiagnosisHeatmapPanel`
  - `ConstructionSimulationPanel`
  - `AcceptanceReportPreview`

### 12.6 交付物建议
即使不写入 Figma，也应准备以下可交付内容：
- 页面帧清单与命名；
- 组件 inventory；
- token 表；
- 每个路由的默认态、loading、error、empty、success 五态；
- 演示稿截图顺序。

## 13. 实现约束与复杂度取舍

### 13.1 必做
- 地图、诊断、施工模拟、验收报告四类主视觉必须明确。
- 工作台信息密度高，但信息块边界清楚。
- 主流程页面之间有清晰跳转与面包屑。

### 13.2 可复用避免新增
- 基础表单与弹层尽量复用 shadcn/ui 风格组件。
- 非必要不创建“超级卡片”“超级容器”之类抽象。
- 图表尽量复用统一标题、图例、说明结构。

### 13.3 新增组件上限建议
- 新领域组件控制在 8-12 个以内。
- 新 token 只为风险、诊断图层、报告样式新增，不单独为某一页造私有 token。

### 13.4 优雅与实现复杂度平衡
- 施工模拟优先 2.5D 剖面，而不是高成本 3D。
- 诊断页优先热力图 + polygon + 剖面切换，而不是复杂体渲染。
- 报告优先 HTML/CSS 可打印版，而不是先做 PDF 引擎专属视觉。

## 14. Demo Polish Checklist

- 首屏出现后 5 秒内能看懂平台定位。
- 地图点位、列表、详情抽屉能在一次点击内串起来。
- 单井详情页同时有照片、指标、雷达预览三类证据。
- AI 诊断过程有步骤感，不是瞬间刷表格。
- 诊断结果同时给等级、评分、病害类型、深度、置信度。
- 推荐方案明确告诉用户“为什么是这个工法”。
- 施工模拟能看到孔位、浆液、锁固、恢复四步。
- 修复前后至少对比高差、平整度、声振、交通开放时间。
- 验收页话术工程化、可量化。
- 所有主页面都具备 loading、empty、error 状态。
- 桌面与移动端不出现文字重叠、图表挤压、按钮遮挡。
- 演示模式能减少非关键信息干扰。

## 15. 待确认项

以下事项需要产品或后续实现阶段确认：
- 地图底图风格是否采用纯矢量道路底图，还是允许更真实的市政底图。
- 是否为路演准备单独的“领导模式”，进一步隐藏技术细节。
- 施工模拟是否需要加入物理模型/视频素材入口，还是只保留前端动画。
- 报告导出优先 HTML 打印、截图导出，还是后续再补 PDF。
- 是否需要单独定义深色演示模式；当前建议先只做浅底工作台，局部深色用于诊断画布。
