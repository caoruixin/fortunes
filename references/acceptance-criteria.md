# Acceptance Criteria

## Demo Flow
1. 用户进入地图页后可看到多个检查井点位，并按风险颜色区分。
2. 用户点击红色高风险检查井后进入单井详情页。
3. 单井详情页展示基本信息、井周照片占位、检测指标、雷达扫描预览和历史维修。
4. 用户可触发 AI 诊断。
5. AI 诊断结果必须包含风险评分、病害等级、病害类型、深度范围、置信度、热力图或 polygon 数据。
6. 系统可生成推荐维修方案，包含推荐工法、材料、注浆孔数量、注浆量、施工时间、开放交通时间。
7. 系统可展示施工模拟，至少包含精准布孔、低压分级注浆、井座锁固与调平、快硬材料恢复四个步骤。
8. 系统可生成“一井一档”验收报告，包含修复前后指标对比和复检建议。

## Engineering
1. `docker compose up --build` 可启动前端、后端和数据库。
2. 后端暴露 OpenAPI 文档。
3. 数据库包含 schema、migration 和 seed 数据。
4. 前端无后端时可使用 mock adapter 演示主流程。
5. API 字段和前端消费字段保持一致。
6. 核心 Mock AI 逻辑为确定性规则，不使用不可复现随机结果。

## Data
1. seed 数据至少包含 20 座检查井。
2. seed 数据覆盖 A/B/C/D 四类病害。
3. 每座高风险 Demo 井至少有检测、诊断、方案、施工、验收的完整数据链路。
4. 坐标数据默认使用虚构或脱敏坐标，不放入真实客户坐标。

## Tests
1. API 测试覆盖检查井列表、单井详情、诊断、维修方案、施工模拟、验收报告。
2. E2E 测试覆盖：地图 -> 高风险井 -> 单井详情 -> AI 诊断 -> 维修方案 -> 施工模拟 -> 验收报告。
3. Mock AI 单元测试覆盖 A/B/C/D 四类病害。
4. 一键验证脚本输出清晰的通过/失败结果。

## Demo Quality
1. 首屏能让客户理解这是道路检查井风险管理和诊修平台。
2. AI 诊断页必须展示“看见地下病害”的信息，不只是普通表格。
3. 修复前后对比必须突出平整度、噪音、沉陷、高差、施工时间和建筑垃圾减少。
4. 验收话术必须可量化，避免无法验证的绝对化承诺。
